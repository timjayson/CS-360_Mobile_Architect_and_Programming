# CS-360_Mobile_Architect_and_Programming
This is my work for the CS-360 class at SNHU


For this course I developed an Inventory Manager mobile application which was designed to help users keep track of inventory items in a simple and organized way. The primary goal of the application was to allow users to securely log into the app, create and manage inventory records, and receive optional SMS notifications when inventory levels became low. The application was intended for small business owners, hobbyists, or anyone looking for a straightforward way to organize inventory without unnecessary complexity. Throughout development, the focus remained on creating an application that was both functional and easy to use.
Several screens and features were necessary to meet these user needs. The application includes a login screen with account creation functionality, an inventory management screen where users can add, edit, and delete inventory items, and SMS notification support for low inventory alerts. When designing the user interface, I tried to focuse on keeping navigation simple and intuitive by using familiar layouts and organizing buttons and information logically. I also paid attention to visual hierarchy so that important actions, such as logging in or adding inventory, were easy for users to find. As development progressed, I made several adjustments to the layout to improve usability after seeing how the application actually looked on an Android device. These changes helped produce a cleaner and more user-centered interface.
When coding the application, I approached the project by breaking it into smaller, manageable tasks instead of attempting to build everything at once. I first created the user interface, then implemented database functionality, followed by the login system and inventory management features. Finally, I integrated SMS permissions and notification capabilities. Building the application incrementally made it much easier to troubleshoot problems as they arose. This structured development process is a strategy I will continue using in future software projects because it allows each feature to be tested before moving on to the next.
Testing was an important part of the development process. I regularly ran the application in the Android emulator after implementing each major feature to verify that everything functioned as expected. I tested account creation, user login, inventory operations, and SMS permission handling. During testing, I discovered several issues involving user interface behavior, permissions, and database interactions that required debugging before the application functioned correctly. This process reinforced the importance of continuous testing throughout development rather than waiting until the end of a project, as it makes identifying and fixing issues much more manageable.
One of the biggest challenges during development involved troubleshooting issues that appeared while integrating multiple components of the application. I encountered problems with emulator performance, Android permissions, and interactions between the user interface and the database. Solving these problems required researching Android documentation, carefully reviewing my code, and testing different solutions until the application behaved correctly. Working through these challenges improved both my technical knowledge and my confidence in debugging Android applications.
